# M2: String Extraction Tool + CI Check Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task.

**Goal:** Build `i18n-extract` — a Rust tool that scans the codebase, identifies user-visible UI strings, wraps them in `t!()`, and extracts unique keys to a YAML file. Also build `check-i18n` CI guard that verifies all `t!()` keys have translations.

**Architecture:** A standalone binary crate `crates/i18n_tools` with two bins: `extract` (scan + rewrite code + generate YAML) and `check` (verify key completeness). Uses regex-based pattern matching (not full syn AST — pragmatic, covers ~80% of cases; remaining 20% handled manually). Reads `.i18n-ignore.toml` for exclusion paths.

**Tech Stack:** Rust, regex, walkdir, serde_yaml

**Reference Spec:** `docs/superpowers/specs/2026-06-19-warp-i18n-zh-cn-design.md` §4

## Task 2: Implement pattern matching engine

The core of the extractor. Identifies UI strings via regex patterns.

**Patterns to match (6 high-frequency patterns):**

```regex
1. .label("X".to_string())           → .label(t!("X").to_string())
2. .with_text_label("X".to_string())  → .with_text_label(t!("X").to_string())
3. Category::new("X",                  → Category::new(t!("X").to_string(),
4. render_body_item_label::<T>("X".to_string(),  → render_body_item_label::<T>(t!("X").to_string(),
5. render_dropdown_item(app, "X", Some("Y"), ...) → render_dropdown_item(app, &t!("X"), Some(&t!("Y")), ...)
6. .span("X".to_string())             → .span(t!("X").to_string())
```

- [ ] **Step 1:** Define regex patterns in `lib.rs`
- [ ] **Step 2:** Implement `apply_patterns(content: &str) -> (String, Vec<String>)` that returns rewritten content + extracted keys
- [ ] **Step 3:** Unit tests for each pattern
